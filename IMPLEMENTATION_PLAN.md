# Unified Ads MCP Platform — Delivery-Ready Implementation Plan

## Executive summary

This converts the PRD and phased plan into an executable delivery structure: a monorepo layout, per-service build order, an Epic → Feature → Task → Subtask hierarchy, milestone gates, parallel workstreams, an immediate scaffolding checklist, the first 30 tickets in execution order, and an agent assignment model.

Hard sequencing rules:
1. No mutation work begins before drafts + policy + approvals exist.
2. No MCP write tools before mutation worker is verified end-to-end.
3. No adapter work touches secrets in plaintext outside encrypted vault helpers.
4. No endpoint ships without org-scoping middleware and an audit emission point.
5. Local dev scaffolding lands before any cloud infra beyond a staging target.

The plan assumes 4–6 AI coding agents working in parallel along clean repo boundaries (web, api, mcp, adapters, workers, infra). Cross-cutting `packages/schemas` and `packages/domain` are owned by a single agent to prevent contract drift.

## Service implementation order

### `packages/schemas` (owner: domain agent)
- **Purpose:** Single source of truth for canonical entity shapes; emits both Zod and Pydantic; drives OpenAPI.
- **Earliest useful version:** Day 1. Zod definitions for `Organization`, `User`, `PlatformConnection`, `PlatformAccount`, `Campaign`, `AdGroup`, `Ad`, `Creative`, `MetricsDaily`, `DraftAction`, `Approval`, `ChangeLog`, `ActivityEvent`, `PolicyRule`, `AgentSession`.
- **Dependencies:** None.
- **Build order:** Types → Zod → Pydantic codegen → OpenAPI emit script → SDK generator.
- **Must-have interfaces:** `@lamani/schemas/{entities,enums,errors,requests,responses}`; `lamani_schemas` Python package mirroring same shapes.
- **Test requirements:** Round-trip serialization parity tests TS↔Python; snapshot of OpenAPI committed.
- **Handoff:** `apps/api` consumes Pydantic models; `apps/web`, `apps/mcp-server`, adapters consume Zod + generated SDK.

### `packages/domain` (owner: domain agent)
- **Purpose:** Pure domain logic (status mapping, diff computation, policy evaluation) free of I/O.
- **Earliest useful version:** Phase 2 — status normalization helpers.
- **Dependencies:** `packages/schemas`.
- **Build order:** Status mappers → diff calculator → policy evaluator → risk classifier.
- **Must-have interfaces:** `normalizeStatus(platform, raw) → CanonicalStatus`; `computeDiff(before, after) → DiffResult`; `evaluatePolicy(action, context) → PolicyDecision`.
- **Test requirements:** 100% unit coverage on diff and policy evaluator.
- **Handoff:** Used by `apps/api` (validation, policy decisions) and adapters (mapping).

### `packages/ui` (owner: frontend agent)
- **Purpose:** shadcn-based components shared across web app.
- **Earliest useful version:** Phase 0 — Button, Input, Table, Sidebar, TopBar primitives.
- **Dependencies:** Tailwind config, design tokens.
- **Build order:** Primitives → layout shell → table → command palette → diff viewer → approval sheet.
- **Must-have interfaces:** Each component exports a typed prop contract; tokens live in `packages/ui/tokens.ts`.
- **Test requirements:** Storybook + axe-core a11y tests on every component.
- **Handoff:** Consumed exclusively by `apps/web`.

### `apps/web` (owner: frontend agent)
- **Purpose:** Operator UI.
- **Earliest useful version:** Phase 0 shell with sidebar + topbar.
- **Dependencies:** `packages/ui`, `packages/sdk`, `packages/schemas` (Zod).
- **Build order:**
  1. Shell + auth pages (Phase 0–1).
  2. Empty primary nav (Phase 2).
  3. Accounts + connect flow (Phase 3).
  4. Campaigns table + detail (Phase 7).
  5. MCP agents settings (Phase 8).
  6. Drafts list + diff (Phase 9).
  7. Approval sheet + activity (Phase 10).
  8. Execute affordance + status (Phase 11).
- **Must-have interfaces:** `useSession`, `useOrg`, `useApi` (typed against SDK); error boundary; theme provider.
- **Test requirements:** Playwright per primary flow; component tests for table, palette, approval sheet.
- **Handoff:** Consumes API only via generated SDK; no direct fetch.

### `apps/api` (owner: backend agent)
- **Purpose:** Source of truth for org-scoped data, drafts, approvals; gateway to workers.
- **Earliest useful version:** Phase 0 health + structlog.
- **Dependencies:** `packages/schemas` (Pydantic), `packages/domain`, Postgres, Redis.
- **Build order:**
  1. Health + structlog + Alembic (Phase 0).
  2. Auth, RBAC, audit, agent sessions (Phase 1).
  3. Read-stub endpoints + OpenAPI emit (Phase 2).
  4. OAuth + token vault + diagnostics (Phase 3).
  5. Sync trigger + sync-job APIs (Phase 6).
  6. Insights + saved views (Phase 7).
  7. Draft endpoints + policy evaluator (Phase 9).
  8. Approval + activity feed (Phase 10).
  9. Execute endpoint + queue dispatch (Phase 11).
  10. OTel + error taxonomy (Phase 13).
- **Must-have interfaces:** `BaseRepo` with org-scoping; `requires_role(*roles)`; `record_audit(...)`; `enqueue(job_type, payload)`.
- **Test requirements:** Cross-tenant denial test on every endpoint; contract tests against OpenAPI snapshot.
- **Handoff:** Web, MCP, workers all read/write through API or shared DB schemas owned by API migrations.

### `apps/mcp-server` (owner: MCP agent)
- **Purpose:** Constrained business-tool surface for AI agents.
- **Earliest useful version:** Phase 0 — `health` tool.
- **Dependencies:** `packages/schemas`, `packages/sdk`, `apps/api`.
- **Build order:**
  1. Skeleton + transports + bearer auth (Phase 0–1).
  2. Read tools (Phase 8): `list_accounts`, `list_campaigns`, `get_campaign`, `get_insights`, `list_drafts`, `get_activity_log`, `sync_account`.
  3. Write tools (Phase 12): `create_draft_pause_campaign`, `create_draft_update_budget`, `approve_draft`, `execute_approved_draft`.
  4. Tracing + per-agent rate limit (Phase 13).
- **Must-have interfaces:** `defineTool({ name, schema, handler })`; agent context resolver; tool-call audit emitter.
- **Test requirements:** Per-tool: schema-violation, cross-tenant, happy path; recorded MCP session test.
- **Handoff:** Calls API exclusively over HTTP with agent bearer; never touches DB or vendor APIs directly.

### `workers/sync-worker` (owner: backend/integration agent)
- **Purpose:** Background ingestion of accounts, campaigns, ad groups, ads, metrics.
- **Earliest useful version:** Phase 6.
- **Dependencies:** `meta-adapter`, `google-adapter`, `apps/api` DB schema, Redis.
- **Build order:**
  1. Worker scaffold + queue connection (Phase 6).
  2. Token refresh job (Phase 3 helper, lifted into worker in Phase 6).
  3. Initial full sync per platform (Phase 6).
  4. Incremental + daily metrics jobs (Phase 6).
  5. Dead-letter + observability (Phase 13).
- **Must-have interfaces:** `JobContext { org_id, account_id, correlation_id }`; idempotent upsert helper keyed on `(org_id, platform, platform_entity_id)`.
- **Test requirements:** Idempotency under retry; rate-limit backoff; partial failure resumes correctly.
- **Handoff:** Writes only to canonical tables; emits `activity_events` rows.

### `workers/mutation-worker` (owner: backend/integration agent)
- **Purpose:** Execute approved drafts via adapter mutations.
- **Earliest useful version:** Phase 11.
- **Dependencies:** `meta-adapter`, `google-adapter`, draft + approval schema, Redis.
- **Build order:**
  1. Scaffold + queue (Phase 11).
  2. Pre-flight re-fetch → execute → post-fetch confirm.
  3. Idempotency keyed on `draft_id`.
  4. Per-org concurrency cap (Phase 13).
- **Must-have interfaces:** `executeDraft(draft) → ChangeLog`; aborts on stale state.
- **Test requirements:** End-to-end with sandbox; stale-state abort; vendor-error → typed `failed`; idempotent retry.
- **Handoff:** Writes `change_logs`, updates `draft_actions.status`, emits `activity_events`.

### `packages/meta-adapter` (owner: integration agent A)
- **Purpose:** Encapsulate Meta Marketing API.
- **Earliest useful version:** Phase 4 — read methods.
- **Dependencies:** `packages/schemas`, `packages/domain` (status mappers), token vault helpers.
- **Build order:** Auth helpers → HTTP client + error taxonomy → `listAccounts` → `listCampaigns` → `getCampaign` → `getInsights` → `pauseCampaign` (Phase 11) → `updateBudget` (Phase 11).
- **Must-have interfaces:** Implements `AdsPlatformAdapter`; preserves `native_payload_json`.
- **Test requirements:** Recorded fixtures (VCR-style) + live sandbox integration; mapping unit tests per canonical field; pagination test.
- **Handoff:** Consumed by sync-worker and mutation-worker only.

### `packages/google-adapter` (owner: integration agent B)
- **Purpose:** Encapsulate Google Ads API.
- **Earliest useful version:** Phase 5 — read methods.
- **Dependencies:** Same as Meta.
- **Build order:** Auth + manager-hierarchy walker → GAQL builder → read methods → mutation methods (Phase 11).
- **Must-have interfaces:** Implements `AdsPlatformAdapter`; manager + child customer support.
- **Test requirements:** Recorded GAQL fixtures + live test account; manager walker test; mutation re-fetch test.
- **Handoff:** Same as Meta.

### `infrastructure` (owner: infra agent)
- **Purpose:** Repeatable local + remote environments.
- **Earliest useful version:** Phase 0 `compose.dev.yml`.
- **Dependencies:** None.
- **Build order:** Local compose → per-service Dockerfiles → CI workflows → staging deploy → terraform for prod (Phase 13).
- **Must-have interfaces:** `make dev-up`, `make migrate`, `make seed`, `make e2e`.
- **Test requirements:** CI runs lint + typecheck + unit + contract on every PR; container build + push on main.
- **Handoff:** All services consume the same `.env.example` keys.

Per-epic objective / value / dependencies / DoD:

**E-00 Monorepo & Developer Foundation** — Objective: deterministic local + CI build for every service. Value: every other epic depends on it. Dependencies: none. DoD: green CI, working compose, staging deploy from main.

**E-01 Identity, Organizations, RBAC** — Objective: multi-tenant isolation + role enforcement + agent identity. Value: blocks every authenticated route. Deps: E-00. DoD: cross-tenant denial proven by tests; agent bearer flow live.

**E-02 Database Schema & Migrations** — Objective: canonical tables + migrations. Value: locks contracts. Deps: E-01. DoD: all PRD tables migrated up/down clean.

**E-03 Shared Domain & Schema Layer** — Objective: Zod + Pydantic + OpenAPI single source. Value: prevents contract drift. Deps: E-00. DoD: TS↔Python parity tests pass; SDK generated from OpenAPI.

**E-04 Next.js Shell & Design System** — Objective: navigable empty UI with all states. Value: unblocks parallel UI work. Deps: E-00, partial E-03. DoD: every primary route renders empty/loading/error.

**E-05 Backend API Foundation** — Objective: routing, middleware, repo base, audit hook. Value: every endpoint inherits safety. Deps: E-01, E-02. DoD: BaseRepo enforces org scope; audit row on every mutation.

**E-06 MCP Server Foundation** — Objective: transports, bearer auth, tool registry. Value: agent surface ready before tools land. Deps: E-01. DoD: `health` tool reachable over both transports under bearer auth.

**E-07 Platform Connections & Token Vault** — Objective: OAuth + encrypted token storage + refresh. Value: gates all platform work. Deps: E-02, E-05. DoD: tokens never leave server boundary; refresh worker keeps connections healthy.

**E-08 Meta Integration** — Objective: Meta adapter (read first, mutate later). Value: half the product. Deps: E-07, E-03. DoD: read methods return canonical data; mutation methods land in Phase 11.

**E-09 Google Integration** — Mirror of E-08.

**E-10 Read-Only Sync Pipeline** — Objective: durable, idempotent ingestion. Value: data appears in product. Deps: E-08, E-09. DoD: connected accounts populate canonical tables on schedule + on demand.

**E-11 Unified Campaigns & Reporting UI** — Objective: internal demo surface. Value: first stakeholder-visible product. Deps: E-10. DoD: filterable unified table, detail page, palette, all states.

**E-12 MCP Read-Only Tool Catalog** — Objective: agent read access. Value: validates MCP design. Deps: E-10. DoD: read tools p95 <500ms, cross-tenant safe.

**E-13 Draft Actions & Policy Engine** — Objective: validated drafts + diffs + policy decisions. Value: safety primitive. Deps: E-10. DoD: pause + budget drafts produce correct diffs and policy verdicts.

**E-14 Approval Workflow & Activity Timeline** — Objective: human checkpoint + audit surface. Value: justifies control plane. Deps: E-13. DoD: approval state machine race-safe; activity feed paginated.

**E-15 Mutation Execution Pipeline** — Objective: execute approved drafts. Value: MVP. Deps: E-14, mutation methods in E-08/E-09. DoD: end-to-end pause + budget update on both platforms with stale-state guard.

**E-16 MCP Write Tools** — Objective: agent close-the-loop. Value: private alpha. Deps: E-15, E-12. DoD: full agent loop scripted in CI.

**E-17 Audit, Logs & Activity** — Objective: end-to-end traceability. Value: compliance + debugging. Deps: spans E-01, E-10, E-14, E-15. DoD: every mutation traceable from agent/user to vendor response.

**E-18 Observability & Operational Readiness** — Objective: logs, metrics, traces, alerts, runbooks. Value: alpha viability. Deps: E-15. DoD: dashboards + SLOs + runbooks committed.

**E-19 Deployment & Environment Management** — Objective: local → staging → prod. Value: ship safely. Deps: E-00. DoD: PR → preview, main → staging, tag → prod.

## Parallel workstreams

After M1, work splits into independent lanes; cross-lane handoffs are explicit:

- **Lane A — Adapters (integration agents A + B):** E-08 read methods ∥ E-09 read methods → contributes to E-10. Later: E-08/E-09 mutation methods → contributes to E-15.
- **Lane B — Frontend (frontend agent):** E-04 → E-11 → E-13 UI → E-14 UI → E-15 UI.
- **Lane C — Platform (backend agent):** E-05 → E-07 → E-10 → E-13 → E-14 → E-15.
- **Lane D — MCP (mcp agent):** E-06 → E-12 (parallel with E-11) → E-16 (after E-15).
- **Lane E — Infra (infra agent):** E-19 staging → E-18 observability → E-19 prod.
- **Lane F — Domain (domain agent):** E-03 evolves continuously; freezes after M3 except additive.

Sequential locks:
- E-15 (mutation execution) blocks on E-14 (approvals) and on adapter mutation methods landing.
- E-16 (MCP writes) blocks on E-15.
- E-10 (sync) blocks on at least one adapter read surface (E-08 or E-09); the other can land later.
- No mutation work touches a real account until E-13 + E-14 are merged on staging.

Allowed parallelism examples:
- E-04 shell can begin while E-01 auth is in progress; shell mocks `useSession`.
- E-03 schemas can lead E-08/E-09 by one phase; adapters consume frozen Zod types.
- E-11 campaigns UI can begin against `metrics_daily` fixtures before E-10 is fully production-grade.
- E-12 MCP read tools can run in parallel with E-11 since both consume the same API surface.

## Sequencing recommendations

Strict ordering rules to enforce in PR review:

1. **No endpoint without org scope.** PR template requires checkbox: "BaseRepo used; cross-tenant test added."
2. **No mutation code before E-13 + E-14 are merged.** CI label gate: PRs touching `workers/mutation-worker` or adapter mutation methods are blocked until both epics are closed.
3. **No MCP write tool before E-15 verified.** Same gate on `apps/mcp-server/src/tools/write/*`.
4. **No token-touching code outside `packages/config` token vault helpers.** Lint rule: forbid `access_token`/`refresh_token` field access outside the vault path.
5. **Schema freezes precede consumer work.** Domain agent tags `@lamani/schemas` versions; consumers pin to a tag.
6. **Sync idempotency proven before scheduling.** E-10 cron schedules disabled in staging until idempotency test suite is green.
7. **Approvals state machine ships with optimistic locking.** E-14 PRs require concurrent-approval test.
8. **Mutation worker re-fetches before and after.** E-15 PRs require stale-state abort + post-mutation confirm tests.
9. **Audit completeness review before alpha.** A pre-M5 task verifies every action emits the right rows in `activity_events`, `audit_events`, `change_logs`.
10. **Observability before alpha traffic.** M5 cannot open until E-18 dashboards + alerts + runbooks are committed.

Hard blockers to expose early:
- Vendor app review (Meta) and developer token approval (Google) — file applications during E-00.
- KMS / secret store provisioning — unblocks E-07; assign to infra agent at M1.
- Production OAuth redirect URIs — needed before staging verifies real tokens.
- Postgres backups + PITR — required before any prod data lands.
